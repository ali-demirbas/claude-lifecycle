All notable changes to this project are documented here. Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/); versioning follows [SemVer](https://semver.org/).

## [Unreleased]

### Fixed

- **WhatsApp's header and button-label ceilings were silently unenforced — `scripts/validate_output.py` hardcoded a shadow copy of `knowledge/channels/*.md`'s own limits, and it never got updated to match.** `whatsapp.md`'s "Hard rules" declare header ≤ 60 chars and button labels ≤ 25, but the validator's `CHANNEL_LIMITS` dict only had a `body` entry for WhatsApp — a header or button field always came back `canon = None`, silently skipping the same Limit/Limit-Inflation check every other channel's fields get. `canonical_limit()` now calls `load_channel_limits()`, which parses every channel file's frontmatter `limits:` block live instead of duplicating it — the channel file is the only source of truth, so this class of drift can't recur. Also closed a second, previously *untested* gap found while verifying the fix: `templates/copy-output.md`'s in-app field set names `Primary CTA`/`Secondary CTA` explicitly, but no channel file declares separate limits for them (every channel declares one `cta` ceiling for both button slots) — `canonical_limit()` now falls back from `primary cta`/`secondary cta` to the channel's plain `cta` limit instead of returning unparsed. Both fixes are locked in with new regression fixtures (`bad_whatsapp_header_copy.md`, `bad_inapp_primary_cta_copy.md`).

### Added

- **`knowledge/channels/` gains a `_template.md` (the one `knowledge/` subfolder that didn't have one) and `scripts/validate.sh` coverage (the one channel-adjacent knowledge category `validate.sh` never checked, alongside brands last pass).** The template extracts the structure all 5 real channel files already share by convention (frontmatter shape, Hard rules, a structure-guidance section, a "when to prefer this channel" table) rather than inventing a new one, and states the actual test for when a new *message type* (rich push, a new in-app widget) is a section in an existing channel's file versus a genuinely new channel needing its own: does it need its own consent tracking and its own frequency-cap line in the portfolio conflict review? New section 9 in `validate.sh` checks required frontmatter keys, `name` matches filename, and the `limits:` block actually contains a parseable ceiling — verified against a deliberately broken fixture before confirming clean on real data.
- **`push.md`'s "Android varies" note now states the actual figure instead of gesturing at it.** Current sources (verified, not assumed stale) put Android's real title/body ceilings noticeably looser than iOS's (title ~65 vs ~40, body ~150-240 vs ~120) — the enforced limits stay at the iOS-conservative floor (this engine has no per-OS send-time targeting to justify a split, and writing to the tighter number never breaks on the looser platform), but the reasoning is now stated rather than left as an unexplained "varies."
- **`knowledge/brands/` gets schema validation — the one knowledge-file category `scripts/validate.sh` never checked.** Skills, journey patterns, industries, and lexicons all have required-key and cross-reference checks; brand configs (the actual per-client data every skill reads) had none, so a half-filled-out template — `tone: <2–3 adjectives>` left as the literal placeholder, a typo'd `industry:` pointing at a sector file that doesn't exist — would load silently and only surface as a confusing downstream symptom in `lifecycle-copy` or `lifecycle-connect`. New section 8 checks: required frontmatter keys present, `name` matches the filename, `industry` (and each declared vertical's `industry`) resolves to a real `knowledge/industries/` file, `channels_live` entries resolve to real channel files, and — the check that would have caught the motivating failure mode directly — no leftover `<placeholder>` token anywhere in frontmatter. Verified against a deliberately broken fixture (wrong name, bad industry ref, bad channel, unfilled placeholder) before confirming it passes clean on the real `demo-b2b-saas.md`; caught and fixed a real bug in the channel-list parser itself along the way (`tr -d '[:space:]'` was deleting the newlines between array entries, gluing `email`+`in_app` into one unmatchable token).
- **Two intake-whitelist answers had no persistence path in the brand config schema, contradicting the schema's own stated purpose.** `lifecycle-intake` always asks primary goal and product rhythm (per its own whitelist), and an existing brand file is supposed to mean "only genuine gaps get asked" — but neither field existed in `knowledge/brands/_template.md`, so both were silently re-asked every session regardless of brand-file history. `product_rhythm` is now a real persisted field (a stable business characteristic — a subscription box's cadence doesn't change quarter to quarter). `default_goal` also persists, but stays a pre-fill, never a silent carry-over: a brand's active priority can shift between engagements in a way tone or incentive policy shouldn't, so it's always re-surfaced for explicit confirmation, same "pre-fills, never replaces" principle already used for website-research and brand-voice-ingestion. `lifecycle-intake`'s own persistence section previously under-described what it writes (omitted `languages`/`markets` from its prose despite them being real template fields) — corrected to the full, accurate list.
- **Multi-vertical brands' known scope boundary is now stated, not silently implied.** `tone`/`formality`/`incentive_policy`/`channels_live` are company-wide even across verticals — already true, but previously left for a reader to infer from what the template didn't mention. Now one explicit sentence names the limitation and the extension point (a per-vertical override block) if a genuinely divergent-tone multi-vertical brand ever needs it — without speculatively building that extension now.

### Added

- **`copy-reviewer`'s banned-word check is now code-assisted, closing a false-negative gap check 1 already recognized but check 2 didn't.** Check 1 (character limits) delegates counting to `scripts/validate_output.py` on the stated reasoning that "code counts faster and more reliably than a model" — but check 2 (banned words) was a pure, unassisted LLM read of a fixed list against every variant and fallback, exactly the kind of exhaustive exact-match task a model is more likely to miss than code. `validate_output.py copy` now accepts `--sector <slug>` and flags every double-quoted literal term from the lexicon's "Banned outright" section and (for email specifically — the one channel with an equivalent section) the channel file's spam-word list, Turkish-casefolded, as a candidate for the reviewer to confirm in context — a `warn()`, not a `fail()`, since a hit can be a false positive (a term inside a longer unrelated word, or inside a required legal disclaimer). Building this surfaced and fixed a real bug before it shipped: a caps-lock Turkish banned word ("KAZANDINIZ") didn't match its lowercase lexicon entry ("kazandınız") because `tr_fold()` fixes İ's Unicode over-expansion but not the reverse ASCII-I-should-fold-to-dotless-ı ambiguity — a new `_banned_match_fold()` collapses i/ı for this comparison only, verified against a new regression test (`scripts/tests/run_tests.sh`) rather than left as an assumption. `copy-reviewer`'s LEGAL verdict is now explicit that it doesn't suppress ordinary FIX findings on the same block. Two smaller additions: a required-fix instruction must name what to write (cite the passing alternative), not just what's wrong; and a borderline naturalness FIX — the checklist's own admittedly least-countable check — gets the same "two judgments to gate" second-pass discipline `evals/rubric.md` already uses elsewhere, balancing the checklist's skeptical-by-default posture against over-rejecting copy that's actually fine.

### Added

- **`copy-writer`/`copy-reviewer` gain a cross-channel voice-cohesion check, closing the one dimension tone calibration never covered.** Tone is already indexed by sector+stage (the lexicon) and by language (the locale overlay), but never by channel — yet email's fuller register and push's 120-character terseness are genuinely different formats, and nothing checked whether a journey moving the same persona through email → push → email still reads as one brand rather than sounding like a different company per channel. Prevention lands in `copy-writer` (a new step 6: read your own blocks in step order, as the user receives them, before returning); detection lands in `copy-reviewer` (a new whole-document pass run once after the per-block checklist) — split the same way the rest of this pair already divides labor, since a single pass checking its own output is a weaker guarantee than an independent adversarial one. Separately, the existing winning-strategy precedent (already informs one variant's starting angle) now explicitly extends to a specific wording that worked, not just an angle label — still re-checked against the current lexicon before reuse, so a phrase that worked for one segment can't smuggle past a banned-word list on precedent alone.

### Added

- **`journey-architect` gains a post-generation self-check, and now reads two sections of its own reference files it was silently skipping.** Every one of the 26 pattern files carries a "Common mistakes" section naming that pattern's specific failure modes (e.g. abandoned-cart's "discounting in step 1 trains the wrong behavior") — the agent's reading list named the pattern file's depth-scaling section as binding but never mentioned this one, so a design could repeat a mistake the repo had already documented by name. Same gap on the industry side: playbooks' Seasonality sections state that seasonal windows modify an existing journey's timing/framing rather than being a separate journey type, but the agent's reading list only named "timing & cadence, segmentation attributes." New step 6 adds a single post-generation reflection pass against the pattern's own mistake list and the existing Rules — calibrated deliberately as Reflection rather than a ReAct-style act/observe loop, since this agent reads static references once and writes once with no external state to re-observe; a `copy-writer`/`copy-reviewer`-style second agent was considered and rejected as disproportionate for one document versus copy's channel-by-channel fan-out. Two smaller fixes: `priority` is now an explicit input (it was implicitly assumed P0, but the doc header requires stating it and the description leaves room for non-P0 delegation too), and numeric/window thresholds in entry or branch conditions now follow one fixed grammar ("≥ N `<event>` in W days") instead of free prose, so `lifecycle-qa`'s boundary-value cases and `lifecycle-audience`'s SQL generation can parse them back out reliably.

### Added

- **`event-analyst` gains persisted output, batched classification, and multi-vertical awareness — closing a gap where `lifecycle-map` never actually reused its analysis.** The agent's own description claimed it was "spawned by lifecycle-connect/lifecycle-map," but `lifecycle-map`'s procedure had zero reference to it and independently reclassified every event from scratch — meaning a 50+ event inventory could get two separate classification passes in one pipeline run, with no guarantee they'd agree on the same event's stage (a real multi-agent consistency risk, not just wasted effort). `event-analyst` now writes its full assessment to `output/<project>/event-analysis.json` (structured note-taking outside the context window, not just a chat message folded into `lifecycle-connect`'s prose and discarded), and `lifecycle-map` now checks for that file first, treating it as a draft to confirm or override rather than a from-zero reclassification. Separately: inventories past ~150 events now classify in batches of ~50 with a carried-forward structured ledger instead of one unbroken pass — the shape of input most exposed to "context rot"/"lost in the middle" accuracy loss on a flat list. Confidence bucketing (Mapped/Assumed/Unmapped) is now pinned to `stage-mapping-rules.md`'s own rule numbers and confidence column instead of being an unstated judgment call. The Mapped table is grouped by stage (feeds `lifecycle-map`'s stage-coverage summary almost directly) rather than one flat list. The agent gained the same `verticals`/`event_prefix` tagging step `lifecycle-connect`/`lifecycle-map`/`lifecycle-journeys` already have — a gap left over from when multi-vertical support was added to those three but not to the agent doing the actual classification work for the largest inventories. A brief self-check step and an optional brand-config input round out the isolation-vs-grounding trade-off inherent to a context-offloaded subagent.

### Added

- **`lifecycle-qa` gains three new case classes, trigger-type-aware guidance, and a coverage matrix.** The case list covered only `event`-shaped triggers, but the journey schema also allows `time` and `segment_entry` triggers — a wait/schedule or a segment recompute has no event to post, so a payload alone was silently implying proof it can't give; a new "Trigger-type awareness" section says explicitly what a `time` case needs (clock advance or shortened wait) and what a `segment_entry` case can and can't prove (a tool's segment-refresh cadence sits outside any payload's control). Three new case classes close concrete edge-case gaps: **boundary pair** (the value just below vs. just at a threshold, not one arbitrary satisfying example — where off-by-one bugs actually live), **timeout/no-response branch** (the no-event side of a wait-gated branch, which needs time simulation, not payload content), and **duplicate delivery** (the same event posted twice, proving no double-entry on a retried webhook — the standard delivery-retry failure mode). A **degraded payload** case (entry event with one real-but-optional param dropped, per the mapped inventory's own optionality) targets the false-positive risk of a suite built from a single idealized sample payload passing while realistic partial production traffic silently fails to enter. Output gains a user-facing coverage matrix (journeys × case classes) — scoped honestly as a generation-coverage summary, not a pass/fail report, since this skill has no execution access to a live CRM.

### Added

- **`lifecycle-audience` gains query-cost safeguards, two new SQL pattern families, and a composable-CDP mode.** The include-with-exclude example was a correlated `NOT EXISTS` with no note that it's a documented cost anti-pattern at scale — `audience-sql.md` now shows the precomputed-exclusion-CTE/anti-join alternative and states when to prefer it. Two new pattern sections: **behavioral sequences** (ordered-event conditions via `LAG()`, for audiences that are a pattern-in-order, not a flat count) and **RFM tiers** (a SQL implementation for the recency/frequency/monetary dimension `ecommerce.md` and the DQS already name but never had a query for). New generator rules close a personalization gap (an identity-only audience list forces the data team to re-derive "which cart" for `{{product_name}}`/`{{cart_url}}` — the query now selects those fields alongside identity) and a type-drift gap (event params can change type mid-window after an SDK update; extraction now falls back across value columns instead of assuming one type holds). Every query now ships with an audience-size signal (dry-run row count, or an explicit instruction to get one) — an unvalidated query is exactly how an audience leak (targeting everyone, or no one) reaches a live send. The CDP input path now recognizes warehouse-native/composable CDPs (reverse-ETL reading the same BigQuery project) as BigQuery mode rather than a redundant trait-JSON translation. `docs/architecture.md`'s pipeline diagram gains the `lifecycle-audience` and `lifecycle-qa` nodes it was missing since both skills were added.

### Added

- **`lifecycle-audit` gains an evidence-coverage discipline, sector-benchmark checks, and a cross-run audit trail.** The audit's 8 dimensions could silently average over however many happened to have evidence, so a portfolio with several "not assessable" dimensions could still read as a healthy score built on a fraction of the picture — the verdict now states scored-vs-total dimensions inline, the same way the DQS states its flags in the score line rather than burying them. Depth-vs-data fit (dimension 3) is now explicitly data-blind without a DQS, closing a fabrication risk where journey shape alone was judged "over-engineered"/"under-leveraged" with no data to back it. Dimension 4 (trigger & exit hygiene) now also compares a journey's actual timing/channel order against its sector playbook's stated cadence guidance, scaled by degree rather than an automatic Critical. A new audit-trail step logs Critical/High findings to `output/<brand>/audit-history.md` and cross-checks the next run against it — a repeat finding is flagged as still-open, a resolved one is acknowledged — the same closed-loop mechanism `winning-strategies.md`/`failed-strategies.md` already established for results and copy. High-stakes verdicts also get a one-line pointer to `evals/rubric.md`'s "two judgments to gate" convention. Brand config templates (`_template.md`, `demo-b2b-saas.md`) gain the `winning-strategies.md` link missed in the prior pass, plus the new `audit-history.md` link.

### Added

- **`winning-strategies.md` — the positive counterpart to `failed-strategies.md`.** The results feedback loop only ever learned what NOT to repeat; a confirmed winning hypothesis vanished after being reported once, so `copy-writer` re-explored from a cold start every time even for a brand/segment with real precedent. `lifecycle-results` now logs confirmed wins in the same format as failures. `copy-writer` treats a matching entry as a starting-point hint for one variant only — the other variant still explores a genuinely different, less-tested angle, so the loop keeps learning instead of converging to a single "winning" angle forever.

### Added

- **Re-entry cooldown and concurrent-journey cap in `lifecycle-journeys`' conflict review.** The existing entry gate only checked *other*, higher-priority journeys' recent sends — nothing stopped a user oscillating near a threshold (e.g. health score dipping in and out of churn-prevention's trigger) from being re-entered into the *same* pattern repeatedly; each individual run was well-formed, but the user experienced relentless re-triggering. A pattern now needs a cooldown (its own duration, or 14 days minimum) after exit before it can re-fire for the same user. Separately, message-volume caps didn't limit how many *distinct* journeys could be simultaneously active for one user — technically-compliant but narratively incoherent (3-4 unrelated asks the same week). Concurrent-journey cap (max 2 non-transactional) now uses the existing precedence order to decide which stays active when the cap would be exceeded.

### Added

- **`lifecycle-intake` gains contextual slot carryover, a CLV "don't know" path, and a plausibility check.** The whitelist only skipped questions answered by connected data or the brand config file, not by the conversation itself — a user mentioning their cadence in passing could still get asked for it formally. Intake now scans the conversation first. The CLV-threshold question had no defined behavior when the user genuinely doesn't know their number (every other question has a stated default); it now degrades to capped-discount-only and records the gap instead of guessing. New plausibility check surfaces (doesn't block) an order-of-magnitude mismatch between a user's quantitative answer and connected data. Multi-vertical question's "event name prefix" reworded with a concrete example, less jargon for a non-technical audience.

### Fixed

- **Rule 3 event-name matching was substring, not token-based.** `stage-mapping-rules.md`'s pattern heuristics (e.g. `use` → engagement) matched anywhere inside an event name, so `user_deleted` or `user_invited` false-positived on `use` and landed in "engagement" instead of being left unmapped or treated as a negative signal. Matching is now token-based (split on `_`/case boundaries, whole-token match), the same discipline `tr_fold` already applies elsewhere in this repo. Prefix patterns like `first_` are unaffected.

### Added

- **Fintech section in `ga4-recommended-events.md`.** Every other major vertical (commerce, games, SaaS, mobile subscriptions) had a dedicated event lookup section; fintech had none, so its events (`kyc_complete`, `first_deposit`, `first_transaction`, etc.) fell through to the less-reliable name-pattern heuristics far more often than other verticals' events did — compounding the Rule 3 bug above for the sector where a false mapping matters most. New section matches `knowledge/industries/fintech.md`'s funnel exactly.
- **Funnel skeleton flags inverted drop-off instead of silently building past it.** `lifecycle-map` step 4 now treats a step with more volume than the step before it as a reported finding (mapping error or data-integrity problem), not as a funnel to present at face value.

### Changed

- **DQS freshness/consistency thresholds are now sector-relative, not fixed.** The 60-day/14-day cutoffs added earlier this cycle were a single universal number across every industry — too loose for a mobile app (real churn signal fires within a week) and potentially too strict for travel (leisure customers may naturally return only once or twice a year; `travel.md`'s own `churn_signal` already says "compute from data" rather than stating a fixed number). Both thresholds now derive from the active industry playbook's `churn_signal` window (freshness = the window itself, consistency = a third of it), with the original 60/14-day figures kept only as the T3/unparseable fallback. Multi-vertical brands apply this per vertical, against each vertical's own `churn_signal`.
- **Volume sufficiency scores off the median sub-period count, not the raw total.** A single anomalous week or month (spike, double-firing bug, bulk import) could previously carry the whole component score even when the rest of the window showed thin, unsustainable volume. The breakdown now flags when one sub-period accounts for most of the total instead of silently letting it inflate the score.

### Added

- **Router disambiguation guardrail.** `skills/lifecycle/SKILL.md` had no rule for a request that plausibly matches two routing-table rows (e.g., "datamı incele" could mean connect's assessment or map's event mapping) — the router would silently pick one. It now states both interpretations and asks, the same "ask, don't guess" principle CLAUDE.md rule 7 already applies to event classification, extended to intent classification. Resolved ambiguity isn't re-asked later in the same session. CONTRIBUTING.md gains a path for contributing back disambiguating phrases.
- **Multi-vertical brand support.** The company config assumed one industry per brand, which breaks for a company that runs genuinely separate product lines (each its own funnel, its own conversion event) — everything was forced through a single blended playbook. `knowledge/brands/<brand>.md` gains an optional `verticals` field (empty by default, so single-industry brands — the common case — are unaffected); when set, `lifecycle-connect` scores funnel completeness and volume per vertical against that vertical's own industry file, `lifecycle-map` tags events to a vertical before stage classification, and `lifecycle-journeys` runs eligibility/prioritization/depth per vertical. Frequency-cap conflict review deliberately stays company-wide: a user in two verticals is still one inbox.
- **DQS hard rules 5–6: freshness and consistency gates.** The five DQS components were all variations of completeness (event diversity, conversion events, funnel, attributes, volume) — nothing checked whether the tracked data was current or stable. A dataset with a rich 12-month history but no events in the last 60+ days, or a primary conversion event that went silent for 14+ days mid-window, could still score a high DQS built on data that no longer describes current behavior. Both gates cap depth at standard regardless of total DQS, following the existing volume-gate pattern (`docs/data-quality-score.md`, `skills/lifecycle-connect/SKILL.md`, `skills/lifecycle-journeys/SKILL.md`).

### Fixed (from the first full real-data run — NovaShop T2-aggregate, all 10 findings of its engine-findings report processed)

- **`validate_input.py` no longer rejects legitimate T2-aggregate input** (finding 1, 🔴): GA4 UI exports carry a report-title preamble before the real header row; the gate assumed line 1 = header and failed 8/8 sheets with bogus `duplicate columns ['']`. Now: header detection (first row in the top 15 with ≥ 2 unique non-empty cells), preamble rows skipped with a warning, only *named* duplicate columns fail, `YYYYMMDD`/`YYYYMM` GA4 date formats parse. New regression fixture `ga4_aggregate_input.csv` locks it in CI.
- **Time-column detection no longer trips on "Realtime"** (finding 2, 🔴): substring regex replaced with word-token matching. Deliberately NOT the suggested `\b` regex — `_` is a word character, so `\btimestamp\b` would silently break the standard `event_timestamp` column; tokens split on non-alphanumerics handle both.
- **`validate_output.py` fails gracefully instead of crashing on a non-dict `est_msgs_per_week`** (finding 3, 🔴), and the per-channel-dict contract is now written where it belongs (`lifecycle-journeys` registry spec). `run_tests.sh` now also treats any Traceback as a test failure regardless of exit code — a validator must fail(), never crash.
- **Cross-group frequency caps** (finding 4, 🟠): the registry gains `audience_overlaps: [[groupA, groupB], ...]`; each declared combo is cap-checked as one merged user (the NovaShop case: new signup + cart abandoner = 9 msgs/week vs cap 8, previously approved silently). Multi-group portfolios without declared overlaps get an explicit warning. Fixture added.
- **DQS activation flag** (finding 5, 🟠): new hard rule 3 in `docs/data-quality-score.md` — design sufficiency ≠ activation sufficiency. When user attributes score 0 (T2-aggregate cap), the DQS carries `activation: blocked (no per-user identity)` alongside the number, `lifecycle-connect` must report it, and the portfolio header template now has a mandatory Activation field. The score itself is not adjusted.
- **Playbook ↔ pattern cross-check in `validate.sh`** (finding 6, 🟡): every prioritized pattern must list that industry in `applicable_industries`, or it's a dead line the eligibility pass never evaluates. The new check found not 1 but **18 dead lines across 8 playbooks**; 17 resolved by extending patterns' `applicable_industries` (each individually defensible — travel referral, mobile-app loyalty via `earn_virtual_currency`, fintech reactivation for never-funded accounts, etc.), 1 (insurance → account-onboarding) resolved by adding insurance to the pattern, deferring to the playbook's documented SME/fleet rationale.
